Overview

The app is cycle-scoped. The active cycle determines which client group, week settings, training maxes, movement profiles, and logged sets are used.

Cycle 1, 2, 3... are separate snapshots TM = 90% of 1RM Work sets are based on TM, not 1RM directly Week 4 is the deload week
The current code stores oneRepMaxesByCycle and trainingMaxesByCycle per client. That means a save in one cycle should not rewrite historical cycles unless a handler explicitly copies forward.

Data Model

Client fields used by the app

oneRepMaxes              current active 1RM by lift
oneRepMaxesByCycle       1RM snapshots by cycle
trainingMaxes            active training max by lift
trainingMaxesByCycle     TM snapshots by cycle
weekAssignmentsByCycle   rep scheme overrides per cycle/week
cycleMembership          explicit list of cycles the client belongs to
movementProfilesByCycle  per-cycle custom movement progressions
movementCalibrationsByCycle calibration status per cycle/lift
loggedSetInputsByCycle   saved workout inputs per cycle/week/lift

What that means

Each client can participate in multiple cycles at once, but the app treats the active cycle as the current working snapshot. Historical cycle records remain available for old recommendations and charts.

When the UI needs a number, it usually asks for the current cycle entry first. If that entry is missing, it falls back to the global field or infers from a prior cycle.

Weekly Flow

Week Rep Scheme Percentages Meaning
Week 1 5 / 5 / 5+ 50 / 60 / 65 / 75 / 85 Base accumulation week
Week 2 3 / 3 / 3+ 50 / 60 / 70 / 80 / 90 Heavier intensity week
Week 3 5 / 3 / 1+ 50 / 60 / 75 / 85 / 95 Top effort week
Week 4 5 / 5 / 5 50 / 60 / 40 / 50 / 60 Deload week

If a cycle marks skipDeloadWeek, the schedule can treat Week 4 differently in the UI, but the normalized baseline still preserves a Week 4 template in the data model.

Formulas

Base TM

TM = round_to_nearest_5(1RM * 0.90)

Implemented by calculateTrainingMax and reused by the profile save flow.

Workout set weight

setWeight = round_to_nearest_5(TM * percentage)

Warm-ups and work sets are all rounded after the percentage is applied.

Top set examples

Week 1 top set = round_to_5(TM * 0.85)
Week 2 top set = round_to_5(TM * 0.90)
Week 3 top set = round_to_5(TM * 0.95)

Calibration estimate

estimated1RM = round(weight * (1 + reps / 30))
rounded1RM   = round_to_5(estimated1RM)
roundedTM    = round_to_5(rounded1RM * 0.90)

After calibration succeeds, needsCalibration is set to false so the same lift does not keep recalculating every logged set.

Important: the live workout calculation clamps a stored TM that is above the actual 1RM back down to 90% of the actual 1RM. That is a defensive guard against stale data, not the primary progression rule.

Cycles and Groups

What moves when a cycle graduates

oneRepMaxesByCycle[nextCycle] = previousCycle.oneRepMaxes
trainingMaxesByCycle[nextCycle] = previousCycle.trainingMaxes + fixed increment
weekAssignmentsByCycle[nextCycle] = { week1: '5', week2: '3', week3: '1' }
cycleMembership = previousMembership + nextCycle

Fixed increments used today

Squat    +10
Deadlift +10
Bench    +5
Press    +5

Custom movements use progressionIncrement from their movement profile instead of the lift defaults.

The current graduation filter only picks clients whose currentCycleNumber matches the active cycle. That prevents mixed-cycle graduation bugs.

Profile progression rule

if hold/calibration is active:
  trainingMax stays the same
else:
  trainingMax = trainingMax + progressionIncrement
oneRepMax stays fixed

The profile modal shows both the manual Actual 1RM editor and the movement recommendation profile cards. The top section writes the cycle snapshot; the lower section displays the movement-level progression values for the selected cycle.

Reset Paths

Manual 1RM save in profile modal

save current cycle oneRepMaxesByCycle[currentCycle]
recalculate trainingMaxes = round_to_5(oneRepMaxes * 0.90)
preserve other cycle snapshots
rebuild future cycle TMs from the edited snapshot

Stall / reset protocol

inspect recent historical records
pick best estimated 1RM
reset current cycle TM from that result
apply forward to later cycles
Existing historical lifts are not rewritten by a reset. The reset changes the cycle snapshot used for future recommendations; past log records remain intact.
